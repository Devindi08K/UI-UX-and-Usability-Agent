RULE 1 — READ THE JSON FIRST (do this before writing a single line of HTML)
================================================================================
The requirements_json contains everything you need. Extract these before starting:

  FROM THE JSON, identify:
  A. screen_name       → used in <title> and header h1
  B. screen_type       → determines layout (see Rule 3)
  C. project_name      → the system this screen belongs to
  D. domain            → the business domain (e.g. "Finance", "Healthcare", "HR")
  E. purpose           → one sentence describing what this screen does
  F. user_role         → who uses this screen (Guest / Staff / Admin / etc.)
  G. key_actions       → list of things the user can DO (these become your buttons)
  H. functional_requirements → the FRs relevant to this screen (drives content)
  I. entity_name       → the PRIMARY data entity on this screen.
                         Derive this from the screen_name and FRs.
                         Examples:
                           "Manage Bookings"     → entity = "Booking"
                           "Transaction History"  → entity = "Transaction"
                           "Patient Records"      → entity = "Patient"
                           "Invoice List"         → entity = "Invoice"
                           "Employee Directory"   → entity = "Employee"

  FROM entity_name AND domain, infer:
  J. realistic_columns → what columns/fields does this entity have?
                         Think about what a real professional system would show.
                         Examples:
                           Booking   → ID, Guest, Room, Check-in, Check-out, Amount, Status
                           Invoice   → Invoice #, Client, Issue Date, Due Date, Amount, Status
                           Patient   → Patient ID, Name, Ward, Doctor, Admission, Condition
                           Employee  → ID, Name, Department, Role, Start Date, Status
                           Transaction → Ref #, Account, Type, Amount, Date, Status

  K. realistic_statuses → what status values does this entity have?
                          Think about the entity lifecycle, NOT hotel statuses.
                          Examples:
                            Invoice   → Draft / Sent / Paid / Overdue
                            Leave Req → Pending / Approved / Rejected
                            Patient   → Admitted / Discharged / Critical
                            Order     → Processing / Shipped / Delivered / Returned
                            User      → Active / Inactive / Suspended

  L. primary_action    → the most important action from key_actions
                         This becomes the button label on each row AND the modal title.
                         Use the EXACT verb from key_actions, not a generic one.
                         Examples:
                           key_actions: ["approve_leave"]   → button: "Approve"
                           key_actions: ["cancel_booking"]  → button: "Cancel"
                           key_actions: ["process_refund"]  → button: "Refund"
                           key_actions: ["deactivate_user"] → button: "Deactivate"

  M. stat_card_labels  → for dashboard/list screens, what KPIs matter?
                         Derive from the domain and FRs, NOT from a hotel template.
                         Examples:
                           Finance  → Total Revenue, Outstanding Invoices, Paid This Month, Overdue
                           HR       → Total Employees, On Leave Today, Open Positions, Avg Tenure
                           Hospital → Admitted Today, Available Beds, Discharged Today, Critical Cases

  // Filter indicator
  document.getElementById('apply-filters-btn')?.addEventListener('click', () => {
    document.getElementById('filters-indicator')?.classList.toggle('hidden');
  });

  // Modal open — reads data attributes to populate modal dynamically
  document.querySelectorAll('.open-modal-btn').forEach(btn => {
    btn.addEventListener('click', () => {
      const modal = document.getElementById(btn.dataset.modal);
      const nameEl = document.getElementById('modal-entity-name');
      const idEl   = document.getElementById('modal-entity-id');
      if (nameEl) nameEl.textContent = btn.dataset.name || '';
      if (idEl)   idEl.textContent   = btn.dataset.id   || '';
      modal?.classList.remove('hidden');
    });
  });

  // Modal close
  document.querySelectorAll('.close-modal-btn').forEach(btn => {
    btn.addEventListener('click', () => {
      btn.closest('.modal-overlay')?.classList.add('hidden');
    });
  });

});

ALL sample data must be derived from the domain and entity_name. Rules:

  - Entity IDs: use a prefix matching the entity.
      Invoice → #INV-2201, Patient → #PAT-3042, Employee → #EMP-1194
      Never use #RES- for non-booking entities.

  - Names: use culturally diverse, realistic names appropriate to the domain.

      Never show YYYY-MM-DD in visible text.

  - Status values: use K (realistic_statuses) — never use hotel statuses
      for non-hotel systems.

  - Stat card values: realistic for the domain.
      Finance dashboard → "$284,500 total revenue" not "120 rooms"

  - Table rows: minimum 6 rows, with VARIED status values across rows.
      Never show all rows with the same status.

  - The primary action button label on each row = L (primary_action).
      A finance system row should say "Approve" or "Process" — not "Cancel Booking".
